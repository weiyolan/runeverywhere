## Goal

Ship the trust-and-reward layer: a server-authoritative badge engine and ISO-week × city leaderboard on top of P4's `points_ledger`; full profile screens (own tab, `user/[id]` with `profile_visibility` enforcement, edit profile); block/report with blocking enforced at the RLS/RPC choke points so a blocked user disappears from every surface; the complete `settings/*` suite including account deletion via an Edge Function; and stranger-safety mechanics — trusted contacts, a tokenized public live-share page that works in any browser, and a compose-SMS-only SOS. Everything stateful lands in migrations `00000000000050`–`00000000000053`; the client never writes points, badges, ranks, or block side effects.

## Workstreams

### A — Migration `00000000000050_blocks_reports.sql`: blocks, reports, every-surface enforcement

One file; all new tables `enable row level security` immediately; all functions `set search_path = ''`.

**A1. Tables + enum.**
- `blocks`: `blocker_id uuid not null references public.profiles(id) on delete cascade`, `blocked_id uuid not null references public.profiles(id) on delete cascade`, `created_at timestamptz not null default now()`, `primary key (blocker_id, blocked_id)`, `check (blocker_id <> blocked_id)`; index `blocks_blocked_idx (blocked_id)`. RLS: SELECT/INSERT/DELETE own rows (`blocker_id = (select auth.uid())`); no UPDATE.
- `create type public.report_reason as enum ('inappropriate_behaviour','harassment','impersonation','safety_concern','spam','other');` (maps 1:1 to the design's sheet: Inappropriate behaviour / Harassment or bullying / Fake or impersonation / Safety concern / Spam or scam / Something else).
- `reports`: `id uuid pk default gen_random_uuid()`, `reporter_id uuid references profiles on delete set null`, `subject_user_id uuid references profiles on delete set null`, `run_id uuid references runs on delete set null`, `message_id uuid references public.messages(id) on delete set null`, `reason public.report_reason not null`, `note text not null default '' check (char_length(note) <= 500)`, `created_at timestamptz not null default now()`; index `(subject_user_id, created_at desc)`. **`set null` (not cascade) on the people FKs so reports survive account deletion** — the safety audit trail outlives the accounts. RLS: INSERT only, `with check (reporter_id = (select auth.uid()) and subject_user_id is not null and subject_user_id <> (select auth.uid()))`. No SELECT/UPDATE/DELETE policies — reports are write-only from clients; triage happens in Studio.

**A2. Helpers (all `security definer stable`, false on null input).**
- `public.is_blocked_pair(p_a uuid, p_b uuid) returns boolean` — `exists (select 1 from public.blocks where (blocker_id = p_a and blocked_id = p_b) or (blocker_id = p_b and blocked_id = p_a))`. The single predicate every surface uses.
- `public.has_blocked(p_blocker uuid, p_blocked uuid) returns boolean` — one-directional check. **Must be definer**: RLS policies cannot subquery `blocks` directly — the blocks SELECT policy only exposes the blocker's own rows, so an inline `exists` inside the `profiles` policy would never see the other user's block row.
- `public.shares_run_with(p_a uuid, p_b uuid) returns boolean` — true when both are approved members of one run, or one hosts a run the other is approved in: two `exists` clauses over `run_members` self-join and `runs` × `run_members` (statuses `'approved'` only).
- `public.can_view_profile(p_profile_id uuid) returns boolean` — the `profile_visibility` gate for the **full profile screen**: self → true; `is_blocked_pair` → false; then by `profiles.visibility`: `everyone` → true; `members` → `shares_run_with(auth.uid(), p_profile_id)` **or** p_profile_id has a `pending` row in a run hosted by the caller (hosts must review requesters); `hidden` → false.

**A3. Surface enforcement — RLS policy patches** (`drop policy … ; create policy …` with the original names; keep original logic, add the block clause — **except #11, which additionally broadens P4's policy**, exactly the work P4 deferred here). This is the phase's hardest correctness item; every read path is listed with its mechanism:

|---|---|---|

**A4. Sever trigger.** `public.handle_block_created()` — `security definer`, AFTER INSERT ON `blocks`:
- Runs hosted by `blocker_id` where `blocked_id` is `pending` → `declined`; `approved` → `removed` (set `decided_at = now()`, `decided_by = new.blocker_id`).
- Runs hosted by `blocked_id` where `blocker_id` is `pending` or `approved` → `cancelled`.
- These UPDATEs deliberately ride P3's membership triggers: conversation membership is deleted, system messages posted, and the blocked side merely sees a normal decline/removal — the block itself is never revealed (A3 #9's `request_declined` exemption keeps that decline visible in B's center, matching the push). Shared runs hosted by third parties are left intact (RLS makes the pair mutually invisible). Unblock (`delete`) has no trigger: severed memberships stay severed; visibility simply returns.

**A5. Acceptance.** `supabase db reset`; in psql with role-played JWTs (P1 smoke pattern): maya blocks marco → maya sees 0 of marco's runs via `runs_within_radius` and `search_runs`, 0 of his `messages`/`run_members`/`notifications` rows; marco selects 0 `profiles` rows for maya; `get_or_create_dm` raises; marco's `join_run` back into maya's run raises "run not open for joining"; marco's pending request into maya's run is now `declined`; rita (no block, no shared run) still reads marco's reviews on his `everyone` profile (the #11 broadening).

**B3. Leaderboard.** Exact view (adjust column spellings per the P4 audit):

```sql
create view public.leaderboard_weekly as
select
  (date_trunc('week', pl.created_at at time zone 'utc'))::date        as week_start,  -- ISO Monday
  to_char(pl.created_at at time zone 'utc', 'IYYY-"W"IW')             as iso_week,
  btrim(p.home_city)                                                  as city,
  pl.user_id, p.display_name, p.avatar_url, p.level,
  sum(pl.points)::integer                                             as points,
  count(distinct pl.run_id) filter (where pl.reason = 'finished')     as runs_count
from public.points_ledger pl
join public.profiles p on p.id = pl.user_id
where p.visibility <> 'hidden'
  and nullif(btrim(p.home_city), '') is not null
group by 1, 2, 3, 4, 5, 6, 7;

revoke all on public.leaderboard_weekly from anon, authenticated;
```

Decisions baked in: **UTC bucketing** (`date_trunc('week')` is ISO-Monday-based in Postgres; the ledger row's `created_at` = completion time); **city = trimmed raw `profiles.home_city`** — exact-string grouping, no fuzzy normalization ("Lisbon" ≠ "Lisboa"; acceptable because one on-device geocoder feeds nearly all values — recorded in Decisions); users without a home city are simply absent. **Plain view, not materialized**: v1 ledger volume is thousands of rows, the aggregate is instant, and block filtering is *per-viewer* — a materialized snapshot cannot encode that, so pg_cron refresh would buy staleness and still need a per-viewer wrapper. The view intentionally runs with owner privileges (aggregates across RLS) which is exactly why direct SELECT is revoked; the only door is:

- `public.get_leaderboard(p_city text default null, p_week_start date default null)` returns `table (rank bigint, user_id uuid, display_name text, avatar_url text, level integer, points integer, runs_count bigint, is_me boolean)` — `security definer stable`. Defaults: caller's `home_city`; current week `(date_trunc('week', now() at time zone 'utc'))::date`. Body: filter view rows by week + city, drop `is_blocked_pair(user_id, auth.uid())`, `rank() over (order by points desc, runs_count desc, display_name)` **after** filtering (viewer-relative ranks), return top 100 plus the caller's own row (union) when ranked below 100. Raise if `auth.uid()` is null; `revoke execute … from anon`.

RLS review notes: every new table default-deny with explicit policies; clients can INSERT only `reports` (self-attributed) and upsert `live_locations` for their own active session, and can DELETE only their own `blocks`/`safety_contacts`; badges/user_badges/points/ranks are written exclusively by `security definer` functions (tier 1). `leaderboard_weekly` deliberately aggregates across RLS as view owner — mitigated by revoking direct SELECT and funneling through `get_leaderboard`, which enforces block filtering and hidden-profile exclusion. `has_blocked`/`is_blocked_pair` are definer because RLS policies cannot read other users' `blocks` rows inline. New client-writable `profiles` columns carry explicit `grant update` statements (C4, D1) — P1 revoked table-wide UPDATE and re-granted a fixed column list, so an ungranted new column fails with `permission denied` despite the own-row policy. Both `alter type … add value` statements are never *used* in their own migration transaction (PG 15 requirement). `supabase db lint` + `npm run db:types` after each migration.

## Risks & gotchas

| Risk | Mitigation |
|---|---|
| A missed read path leaks a blocked user (hardest correctness item) | Enforcement lives at the four RLS choke points (`profiles`/`runs`/`run_members`/`messages`) + the definer patches (`get_run_by_invite`/`get_or_create_dm`/`list_conversations`/`join_run`) — not per-screen; A3's table is the audit checklist; H2 smoke + step 6 walk every surface; budget ~3 days for A alone |
| RLS policy subquery on `blocks` silently sees no rows (blocks RLS hides other users' rows) | Never inline-query `blocks` in a policy — always the definer helpers `has_blocked`/`is_blocked_pair` (A2); smoke test asserts the profiles direction explicitly |
| Broadcast delivers blocked senders' messages (topic-level auth can't filter rows) | Accepted: client-side drop via `useBlocks` set (E2); history is RLS-clean on next fetch; document as known limit |
| `leaderboard_weekly` view bypasses RLS by design | Direct SELECT revoked from anon/authenticated; only `get_leaderboard` (definer, block-filtered, hidden-excluded) reads it; smoke test asserts the revoke |
| P4 interface drift beyond the audited names | Preconditions audit before Workstream B; the two known traps are already encoded in this plan (`run_tracks.distance_m` is **meters** — convert `/ 1000.0`, never just rename; the level name column is `levels.title`); anything else that differs → fix the plan first, don't substitute blindly |
| `alter type … add value` used in same transaction → error | Both migrations only *reference* new kinds inside function bodies executed later; never insert rows with them in the same file |
| UTC week bucketing: late-Sunday local runs land in "next week" | Accepted v1 (single global convention beats per-user timezones); client computes the toggle's `week_start` with the same UTC rule so UI and SQL agree |
| City string mismatch splits a leaderboard ("Lisboa" vs "Lisbon") | Accepted v1: one geocoder feeds `home_city`; exact-trim grouping recorded; revisit with a cities table if metrics demand (P7) |
| Dawn/dusk badges use longitude/15 timezone approximation | Documented in `run_local_hour`; worst case ±1 h at zone edges — cosmetic for badge thresholds |
| `expo-sms` unavailable (simulator, iPad, no SIM) | `SMS.isAvailableAsync()` gate with share-sheet fallback (G2); QA on SIM devices |
| App Store safety-claim rejection (PLAN.md §6) | SOS is compose-only; reconciled copy in G3/G4 states "never contacts emergency services"; add to P7 reviewer notes |
| Share token leaked/forwarded | 32-hex unguessable token, 12 h expiry, ended on finish, one active session per user, page holds only name + latest point; user can end anytime |
| `delete-account` partial failure (storage removed, auth delete fails) | Order: storage first, auth delete last (retryable — rerun completes); function is idempotent per step; reports survive by FK design |
| Rating aggregates still count blocked reviewers while their reviews are hidden | Accepted: aggregates are global server truth; recomputing per-viewer is disproportionate. Noted so QA doesn't flag the mismatch |
| Roster-derived counts drift for viewers with blocks (hidden member rows) | Capacity truth stays in `join_run`/`respond_to_join_request`/`run_approved_count` (definer, unfiltered); UI counts may read one lower for the blocker — cosmetic |
| Award trigger fires on every ledger insert | A handful of indexed counts per completion (B2 ensures indexes); revisit only if P7 load tests care |

## Decisions made by this plan

- **Leaderboard route = new root route `/rewards`** (PLAN.md §4 lists none): matches the design's "Rewards" screen (level + badges + leaderboard as one surface) reached from the Profile tab, parallel to the existing root `notifications` route.
- **`leaderboard_weekly` is a plain view, not materialized**: v1 volume makes the aggregate instant, and block filtering is per-viewer — impossible to bake into a snapshot; no pg_cron refresh to operate.
- **Week bucket = `date_trunc('week', created_at AT TIME ZONE 'utc')::date`** (ISO Monday, UTC) + `IYYY-"W"IW` label; client mirrors the same rule.
- **City normalization = `btrim(home_city)` exact grouping**, no fuzzy matching; users without `home_city` excluded.
- **View access via `get_leaderboard` RPC only** (revoke direct SELECT): the block filter and own-rank append live there; ranks are viewer-relative (computed after filtering).
- **Badge criteria live in `award_badges` (single definer function), catalog table is display-only**; triggers on `points_ledger` + `reviews` inserts; idempotent via the `user_badges` PK.
- **Dawn/dusk = longitude/15 local-hour approximation** (`run_local_hour`); thresholds: before 07:00 / at-after 19:00.
- **Block visibility is asymmetric at the profile-row level**: whoever blocked you is fully unreadable to you; whoever *you* blocked stays row-readable (blocked list + design's banner) while every content surface filters pairwise via `is_blocked_pair`.
- **Blocking severs live relationships** (auto decline/remove/cancel through existing member-status transitions, so P3 triggers handle chat + notifications) and is never disclosed — the blocked side sees ordinary declines (their center row and push both survive: A3 #9 exempts `request_declined` from the actor-block filter, and `join_run` re-requests fail with the generic "run not open for joining"); unblock does not restore severed memberships.
- **Group chats containing a blocked third party stay usable**; only DMs refuse sending; messages between the pair are mutually invisible.
- **`profile_visibility='members'` is enforced at the full-profile level** (`can_view_profile` RPC + limited card), not by hiding the profile row — hiding rows would break host cards and rosters for browsers; `members` includes hosts reviewing pending requesters. `hidden` remains row-level (P1 policy semantics kept).
- **`reports` are client-write-only** with `on delete set null` people FKs so the safety trail survives account deletion; triage in Studio (no admin UI in v1); reasons enum mirrors the design's six options.
- **Notification prefs = `profiles.notification_prefs jsonb`** (P5 may not create extra tables per PLAN.md §3); keys `requests/accepts/messages/reviews/rewards`, absent = on; **prefs gate push only**, the center always records. The `reviews` key gates both of P4's review-adjacent kinds — `run_completed` (the rate-the-crew prompt) and `review_received` (the "rated you" push `submit_review` fans out; P4 D3 marks the name a do-not-rename cross-phase contract). `notification_pref_key` returns `null` for unmapped future kinds, leaving them deliberately ungated. Design's "Weekly leaderboard" toggle relabeled "Rewards & leaderboard" to also cover `badge_earned`.
- **Weekly leaderboard push ships in P5** (cron Mon 09:00 UTC) — it is the only thing that makes the design's toggle real.
- **`like_types run_type[]` column added to profiles** — the design's "Run types you like" had no schema home.
- **Trusted-contact limits**: partial unique index for the single emergency contact + BEFORE trigger for the ≤5 trusted rule; contacts are manual name/label/phone entry (no `expo-contacts` — one fewer native module).
- **Live share = hex(16-byte) token, 12 h expiry, one active session per user, latest-point-only `live_locations` row (PK = session)**; page is server-rendered HTML + 15 s JSON polling + OSM embed iframe + Google Maps link — no map SDK, no websockets.
- **Client writes `live_locations` directly under RLS** (own active session), throttled to 20 s inside the existing P4 background task; Edge Function reads with service role (contacts have no accounts).
- **SOS = press-and-hold → prefilled SMS composer only** (expo-sms), auto-starting a share session for the link; the prototype's "call services" copy is removed everywhere (PLAN.md §6).
- **Design's "Share my next run" button cut**; the auto-share toggle + in-run share action cover the need with less state.
- **Favorites cut from v1 entirely** — the profile favorite heart is dropped (`favorites` is `(user_id, run_id)`; a person-favorite has no schema home and P5 may not add tables beyond its list), and the run-detail heart P2 deferred here ("earliest P5") is not picked up either: no shipped surface lists favorites and PLAN.md assigns none, so the core `favorites` table stays dormant through v1 and P7's route/state audit can skip it.
- **"Connected apps" settings section deferred to P6** (owns `connected_accounts`); the hub gains the section then.
- **Rank ties break by `runs_count desc, display_name`** — deterministic UI ordering.
- **Push channel `rewards` added** for `badge_earned`/`leaderboard_weekly`, deep-linking to `/rewards`.
